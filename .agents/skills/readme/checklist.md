# README Quality Checklist

Run through every item before finalizing a README. Items are grouped by category.

## Four questions test

- [ ] A reader can quickly determine what this project does
- [ ] A reader can quickly determine how to use or install it
- [ ] A reader can find who made it and how to contact them
- [ ] A reader can find links to learn more (docs, community, issues)

## Project type alignment

- [ ] Project type correctly identified (library, application, CLI, framework, starter, collection)
- [ ] README structure follows the format for that project type (see formats.md)
- [ ] Sections are in the recommended order for the project type
- [ ] Key emphasis matches the audience (code for libraries, screenshots for apps, commands for CLIs)

## Content quality

- [ ] Tagline is one sentence and front-loads the problem solved
- [ ] Highlights list has 3-6 items maximum
- [ ] Overview is 1-2 paragraphs, not a wall of text
- [ ] Usage examples are minimal, realistic, and runnable
- [ ] Code blocks use syntax highlighting
- [ ] Installation shows the simplest method first (package manager one-liner)
- [ ] Dev setup is NOT in the main install section (belongs in CONTRIBUTING.md or bottom)
- [ ] No full API documentation in the README (link to docs instead)

## Conciseness

- [ ] Every section earns its place — no filler or placeholder content
- [ ] Template sections without real content have been removed
- [ ] Long lists use `<details>` to stay scannable
- [ ] No unnecessary preamble or throat-clearing
- [ ] README is as short as it can be without being any shorter

## Visual elements

- [ ] Projects with visual output have at least one screenshot or GIF
- [ ] Screenshots are near the top, not buried below
- [ ] Code examples show expected output where it helps
- [ ] Badges are selective and each provides real value
- [ ] Images have alt text

## Freshness

- [ ] All links work (docs, badges, screenshots, external references)
- [ ] Referenced commands and features actually exist
- [ ] Installation instructions are current
- [ ] Screenshots reflect the current state of the software
- [ ] Version requirements and compatibility info are accurate

## Supporting files

- [ ] README links to CONTRIBUTING.md for dev setup (if applicable)
- [ ] README links to LICENSE
- [ ] CHANGELOG.md exists for versioned projects (if applicable)
- [ ] All supporting files referenced in the README actually exist

## Writing quality

- [ ] Written for the target audience (not too technical for end users, not too basic for developers)
- [ ] Consistent terminology throughout
- [ ] No unexplained jargon (or audience is expected to know it)
- [ ] Professional, inviting, and approachable tone
- [ ] Grammar and spelling are correct
